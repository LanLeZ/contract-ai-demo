## 合同智能解读系统架构分层说明

本文档总结当前系统的整体分层设计，方便架构沟通与后续演进。系统自上而下可分为五层：

- **表现层（前端 UI 展示层）**
- **前端应用与交互层（客户端业务逻辑层）**
- **接口与安全层（HTTP API / 认证网关层）**
- **领域业务与智能服务层（合同解析 / RAG 问答 / 合同对比 / 知识图谱）**
- **数据与存储层（关系数据库 / 向量库 / 文件存储）**

---

## 一、表现层（前端 UI 展示层）

**主要职责**

- 向用户展示登录注册、首页、合同列表与详情、智能问答对话框、知识图谱可视化等界面。
- 负责基础交互布局和视觉呈现，不直接处理复杂业务逻辑。

**技术栈**

- `React`：组件化视图库。
- `react-router-dom`：单页应用路由与导航。
- `Ant Design`：UI 组件库（布局、按钮、表格、Tabs、Modal 等）。
- `echarts-for-react` + `ECharts`：知识图谱力导向图可视化。

**代表模块**

- 页面组件：`pages/Login.jsx`、`Register.jsx`、`Dashboard.jsx`、`Contracts.jsx`、`PersonalCenter.jsx`。
- 全局入口：`App.jsx`（路由与页面挂载）、`main.jsx`，全局样式 `App.css` / `index.css`。

**在系统中的作用**

- 是用户看到的“界面层”：负责展示合同文本、问答对话、知识图谱三元组等信息。
- 将用户操作（上传文件、提问、点击会话、查看知识图谱等）转化为对前端业务层的调用，间接触发后端 API。

---

## 二、前端应用与交互层（客户端业务逻辑层）

**主要职责**

- 管理前端的认证状态、路由跳转、HTTP 调用与加载状态。
- 封装与后端交互的客户端“服务层”，提供统一的 API 调用接口。
- 处理通用错误（如 401 失效自动跳转登录）、消息提示等。

**技术栈**

- `React Context` + 自定义 Hook：实现前端登录态与用户信息管理。
- `axios`：HTTP 客户端，统一配置 baseURL 和拦截器。

**代表模块**

- 认证上下文：
  - `contexts/AuthContext.jsx`：保存 `user`、`token`、`isAuthenticated` 等状态。
  - `hooks/useAuth.js`：对 UI 层暴露 `login` / `logout` / `getCurrentUser` 等能力。
- 前端服务封装：
  - `services/api.js`：封装 axios 客户端，配置 `VITE_API_BASE_URL`、请求/响应拦截器、自动附带 JWT。
  - `services/auth.js`：注册、登录、获取当前用户、更新个人资料。
  - `services/contracts.js`：上传合同/图片合同、合同列表与详情、删除、合同问答、会话列表与历史、知识图谱获取与抽取。

**在系统中的作用**

- 是前端的“业务逻辑/应用层”：对上为 UI 层提供简洁的函数接口（如 `contractService.uploadContract`、`contractService.contractQA`），对下统一调用后端 REST API。
- 负责维持一致的前端交互体验（loading、错误提示、登录态校验等），让表现层只关注界面布局与展示。

---

## 三、接口与安全层（HTTP API / 认证网关层）

**主要职责**

- 对前端暴露统一的 RESTful API。
- 处理路由分发、参数校验、认证授权检查（基于 JWT）。
- 将 HTTP 请求适配为内部业务服务调用，并将结果封装为 JSON 响应。

**技术栈**

- `FastAPI`：后端主框架。
- `Pydantic`：请求/响应数据模型定义与校验。
- `fastapi.middleware.cors.CORSMiddleware`：跨域访问配置，允许前端域名访问。
- 安全与认证：基于 JWT 的认证与依赖注入。

**代表模块**

- 应用入口与路由挂载：
  - `app/main.py`：创建 FastAPI 应用，注册 CORS 中间件与各业务路由（`auth`、`documents`、`search`、`qa`、`kg`、`compare` 等），提供 `/api/health` 健康检查。
- 认证与用户：
  - `app/api/auth.py`：用户注册、登录、获取当前用户信息、个人资料修改。
  - `app/security.py`：JWT 生成与校验、`get_current_user` 依赖。
- 业务 API 路由：
  - `app/api/documents.py`：合同文件/图片上传、合同列表、详情、删除。
  - `app/api/qa.py`：合同智能问答与会话历史接口（如 `/api/documents/{id}/qa` 及 sessions）。
  - `app/api/search.py`：向量检索/全文检索相关接口。
  - `app/api/kg.py`：合同知识图谱三元组的抽取与查询接口。
  - `app/api/compare.py`：合同对比任务创建与结果查询。

**在系统中的作用**

- 是前后端的“边界层”和“安全门”：所有来自浏览器的调用都从这里进入，先经过 JWT 校验与参数校验，再进入领域服务层。
- 对上隐藏内部实现细节，仅暴露稳定的 REST API；对下负责把 HTTP 请求转换为对业务服务的调用。

## 四、领域业务与智能服务层

（合同解析 / RAG 问答 / 合同对比 / 知识图谱）

**主要职责**

- 实现系统的核心业务逻辑与智能处理能力，包括：
  - 合同与图片文件的解析与文本抽取。
  - 文本分块与元数据标注，供向量化与检索使用。
  - 合同智能问答（RAG：向量检索 + 大模型生成）。
  - 合同条款级对比与风险分析。
  - 合同知识图谱三元组抽取。

**技术栈**

- Python 业务服务类/函数，按领域拆分在 `app/services/` 下。
- 文本切分：基于 `langchain.text_splitter` 的自定义切分器（Markdown/法律/合同等）。
- 大语言模型：阿里云 `DashScope` SDK 调用通义千问，用于 RAG 回答与对比分析等。
- 日志：统一使用 `logging` 记录业务流程与异常。

**代表模块**

- 合同解析与切分：
  - `app/services/document_parser.py`：根据文件类型与内容进行解析（PDF / DOCX / Markdown / TXT / 图片 OCR 等），返回统一的合同文本及元数据。
  - `app/services/text_splitter.py`、`app/services/contract_splitter.py`、`app/services/legal_splitter.py`：
    - 封装法律条文和合同文本的切分策略（按标题、条款编号、句子等）。
    - 输出带元数据的分块（如 `clause_marker`、页码等），供后续向量化与检索。
- RAG 问答与 LLM 调用：
  - `app/services/llm.py`：
    - `QwenChatClient`：封装通义千问调用，支持普通对话、问题范围与法律名称分析（`analyze_scope_and_laws`）、基于上下文的 RAG 问答（`answer_question_with_rag`）。
    - `attach_contract_compare_llm_analysis`：对合同条款差异做重要性与风险说明。
  - `app/api/qa.py`：
    - 串联合同归属校验 → 用 `VectorStore.search` 检索相关合同片段和法律条文 → 调用 `QwenChatClient.answer_question_with_rag` → 记录 `Conversation` 历史。
- 合同对比：
  - `app/services/contract_compare.py`：
    - 使用 `ContractTextSplitter` 将左右两份合同切成条款（含 `clause_marker`）。
    - 对齐条款编号，区分 `only_in_left` / `only_in_right` / `changed_clauses`，生成结构化对比结果。
    - 进一步调用 LLM 为每个差异条款生成“重要性 + 风险分析”说明。
- 知识图谱抽取：
  - `app/services/kg_extractor.py`（结合 `models.KnowledgeTriple` 与 `api/kg.py`）：
    - 对合同文本调用大模型或规则抽取 (head, relation, tail) 三元组。
    - 将结果写入 `knowledge_triples` 表，供前端查询与可视化。

**在系统中的作用**

- 是整套系统的“核心大脑”和“领域服务层”：真正实现“合同智能解读”的业务目标。
- 将底层的大模型、向量库能力封装为高层业务语义：上传文件 → 合同片段向量化；提问 → 带证据的回答；选两份合同 → 结构化差异 + 风险评估；查看知识图谱 → 实体关系可视化。

## 五、数据与存储层（关系数据库 / 向量库 / 文件存储）

**主要职责**

- 负责所有持久化存储，包括：
  - 用户与合同元数据。
  - 合同条款分块与向量索引。
  - 合同问答会话与历史消息。
  - 合同知识图谱三元组。
  - 原始上传文件与中间产物（图片等）。

**技术栈**

- 关系数据库：`MySQL`，通过 `SQLAlchemy` ORM 访问。
- 向量数据库：`ChromaDB` 持久化存储，封装在自定义 `VectorStore` 中。
- 向量计算：`DashScopeEmbedder` 调用 DashScope 嵌入模型，作为 Chroma 的自定义 EmbeddingFunction。
- 文件系统：本地目录 `uploads/`（合同文件）与 `chroma_db/`（向量库持久化）。

**代表模块**

- 数据库访问：
  - `app/database.py`：从环境变量读取 MySQL 配置，创建 `engine`、`SessionLocal`、`Base`，提供 `get_db()` 依赖。
  - `app/models.py`：定义 ORM 模型，例如：
    - `User`：用户信息。
    - `Contract`：合同元数据、解析的文本内容、切分数量、分类等。
    - `ContractChunk`：合同切分片段及其元数据。
    - `Conversation`：问答会话历史。
    - `ContractCompare`：合同对比任务与结果 JSON。
    - `KnowledgeTriple`：知识图谱三元组。
- 向量库与嵌入：
  - `app/services/embedding.py`：`DashScopeEmbedder`，封装 `embed_documents` / `embed_query`。
  - `app/services/vector_store.py`：
    - 延迟导入 `chromadb`，使用 `PersistentClient` 建立持久化集合。
    - 定义 `DashScopeEmbeddingFunction` 作为自定义嵌入函数。
    - `add_documents()`：对切分好的 chunks 批量向量化并写入集合。
    - `search()`：基于向量的相似度检索，支持 `filter_metadata`（如 `user_id`、`contract_id`、`source_type` 等）。
- 文件存储：
  - `uploads/` 目录：`api/documents.py` 中按用户创建子目录保存上传的合同文件/图片。
  - `CHROMA_PERSIST_DIR` 或默认 `./chroma_db`：Chroma 向量库持久化目录。

**在系统中的作用**

- 是整个系统的数据基础层：保证用户、合同、对话、知识图谱与向量索引的长期存储与一致性。
- 为上层业务提供统一的数据访问接口（ORM + VectorStore），业务代码只需要关心“存什么/查什么”，而不必关心底层存储细节。

## 典型调用链示例（文字版）

- **合同上传与解析**
  - 前端 `Contracts.jsx` 调用 `contractService.uploadContract` →
  - 通过 `services/api.js` 发送 `POST /api/documents/upload` →
  - `app/api/documents.py` 校验用户与参数、保存文件 →
  - 调用 `document_parser` 与文本切分器生成 `ContractChunk` →
  - 使用 `VectorStore.add_documents` 写入 Chroma 向量库 →
  - 在 MySQL 中写入 `Contract` 与 `ContractChunk` 记录。

- **合同智能问答（RAG）**
  - 前端在 `Contracts.jsx` 的“智能问答” Tab 中调用 `contractService.contractQA` →
  - `POST /api/documents/{id}/qa` →
  - `app/api/qa.py` 校验合同归属，使用 `VectorStore.search` 检索相关合同片段和法律条文 →
  - 调用 `QwenChatClient.answer_question_with_rag` 生成回答 →
  - 将问答轮次写入 `Conversation` 表，并返回答案给前端。

- **知识图谱查看**
  - 前端切换到“知识图谱” Tab，通过 `contractService.getContractKG` 调用 `/api/contracts/{id}/kg` →
  - 如无数据，则触发 `extractContractKG` 调用 `/api/contracts/{id}/kg-extract` →
  - 后端使用 `kg_extractor` 和 LLM 生成三元组写入 `KnowledgeTriple` →
  - 前端使用 `echarts-for-react` 将三元组渲染为力导向图。
