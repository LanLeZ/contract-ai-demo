## 开发计划（Day 3 - Day 10）

### 第3-4天：文件上传与合同管理
- [ ] 文件上传API（PDF/DOCX），支持多文件上传与基础校验（大小 / 类型）
- [ ] 文件解析服务（PDF → 文本，DOCX → 文本），统一抽象为 `DocumentParser` 接口
- [ ] 合同元数据入库（关联用户、文件名、存储路径、上传时间等）
- [ ] 合同列表页面：分页展示、按时间/名称排序、基本搜索
- [ ] 合同删除功能：软删除优先，确保不会影响历史对话记录

### 第5-7天：RAG核心功能实现
- [ ] 嵌入服务与模型选择（如 `text-embedding-3-large`），封装为独立的 `EmbeddingService`
- [ ] 向量数据库（Chroma）初始化与配置，确定集合命名规范与 metadata 结构
- [ ] 法律条文知识库构建：批量导入法律条文 Markdown，区分 `source_type=legal`
- [ ] 合同向量化：对用户上传合同进行分段切分与向量化，标记 `source_type=contract`
- [ ] 向量检索服务：封装检索接口，支持按 `source_type` / `user_id` / `contract_id` 过滤
- [ ] 文本切分质量验证脚本（`verify_text_splitting.py`）用于调参与质量回归

### 第8-10天：智能问答与对话体验
- [ ] RAG 问答流程编排：检索法律条文 + 目标合同片段，构造高质量 Prompt
- [ ] LLM 回答服务：封装为后端 API，支持流式输出与错误重试
- [ ] 前端问答界面：支持选择合同 / 不选合同时的“纯法律咨询”模式
- [ ] 对话历史保存：将问题、回答、关联合同与检索片段摘要入库
- [ ] 对话历史列表与详情页：支持按合同 / 时间筛选与查看
- [ ] 基础监控与调试：简单的日志、检索命中数统计、LLM 调用失败率监控

### 后续优化方向（可选）
- [ ] 多模型支持与模型切换（本地/云端）
- [ ] 合同风险项自动摘要与标签化
- [ ] 多语言支持（中/英合同混合场景）
- [ ] 角色与权限控制（团队版、多用户共享知识库）
