## 第2章 相关技术与理论基础
- **2.1 B/S 架构与前后端分离技术**
  - **2.1.1 B/S 架构概述**  
    B/S（Browser/Server）架构以浏览器作为客户端承载主要交互功能，服务端负责业务逻辑处理与数据存储。该架构具有部署维护成本低、跨平台能力强、易于迭代更新等特点，适用于面向多用户的在线信息系统。  
  - **2.1.2 前后端分离思想**  
    前后端分离将 UI 展示与交互逻辑（前端）和业务处理与数据访问（后端）进行解耦。前端通过 HTTP/JSON 调用后端 REST API 完成功能，实现“接口驱动开发”。其优势体现在：  
    - 前后端可并行开发，提升协作效率；  
    - 通过统一接口规范降低耦合度，便于后期扩展与重构；  
    - 前端可独立做性能优化与体验优化，后端可独立做安全与并发优化。  
  - **2.1.3 在本系统中的应用**  
    本系统采用 B/S 架构与前后端分离模式：前端负责合同管理与问答交互界面，后端负责合同/法律文本处理、向量检索、调用大语言模型生成答案，并将结果以 API 形式返回给前端展示。
- **2.2 Web 开发框架 FastAPI 技术**
  - **2.2.1 FastAPI 概述**  
    FastAPI 是基于 Python 的现代 Web 框架，底层依赖 Starlette（ASGI 异步网络框架）与 Pydantic（数据校验与序列化）。它支持使用类型注解声明请求参数与返回值结构，并可自动生成 OpenAPI 文档，适合构建规范化的 RESTful API 服务。  
  - **2.2.2 关键特性**  
    - **高性能**：支持异步 I/O 与并发处理，适合高频接口调用场景；  
    - **强类型与自动校验**：参数与响应模型清晰，减少脏数据与接口误用；  
    - **自动接口文档**：降低前后端联调成本，提升开发效率；  
    - **工程化友好**：可配合依赖注入、中间件、路由分层组织业务模块。  
  - **2.2.3 选择 FastAPI 的原因**  
    本系统需要提供合同上传、法律知识库检索、智能问答等接口，并可能在问答阶段触发向量检索与大模型调用，存在一定并发压力。FastAPI 在性能与开发效率之间取得较好平衡，能够满足本系统快速迭代与稳定运行需求。  
  - **2.2.4 在本系统中的作用**  
    FastAPI 用于实现后端接口层，负责：请求校验、业务逻辑编排（检索→拼接上下文→模型调用）、统一错误处理、日志记录以及将处理结果返回给前端。
- **2.3 前端框架 React 技术**
  - **2.3.1 React 概述**  
    React 是一种用于构建用户界面的前端框架，采用组件化与声明式编程思想。其核心是“状态驱动视图”，通过虚拟 DOM 与 Diff 算法提高 UI 更新效率，适合构建交互频繁的单页应用（SPA）。  
  - **2.3.2 关键特性**  
    - **组件化复用**：将合同列表、上传表单、问答对话框、引用展示等拆分为组件，便于复用与维护；  
    - **路由管理**：通过 React Router 实现不同功能页面间的无刷新切换；  
    - **工程化生态**：可与状态管理、请求库、UI 组件库协同，提升开发效率。  
  - **2.3.3 UI 组件库（Ant Design）的作用**  
    Ant Design 提供表格、表单、上传、弹窗等成熟组件，能够快速搭建合同管理与问答交互界面，并保证视觉一致性与交互规范性。  
  - **2.3.4 在本系统中的应用**  
    前端负责呈现合同上传/管理、合同预览、问答对话、引用条款展示、历史记录等界面，并通过 Axios 等方式调用后端 API 实现数据交互。
- **2.4 数据库 MySQL 与 ORM 技术**
  - **2.4.1 MySQL 概述**  
    MySQL 是典型的关系型数据库管理系统，支持结构化数据的事务处理与一致性约束，适合存储用户、合同元数据、问答历史等具有明确字段结构的数据。  
  - **2.4.2 ORM 技术概述**  
    ORM（Object-Relational Mapping）用于将数据库表映射为程序中的对象模型，实现面向对象的数据操作。与直接编写 SQL 相比，ORM 能降低样板代码，提高可维护性，并减少因手写 SQL 导致的注入风险。  
  - **2.4.3 在本系统中的数据存储职责**  
    - **结构化数据（MySQL）**：用户信息、合同基本信息、上传记录、问答会话与审计日志等；  
    - **非结构化语义索引（向量库）**：合同片段向量与法律条文片段向量。  
    通过“关系库 + 向量库”协同，使系统既能完成业务管理，也能完成语义检索与智能问答。
- **2.5 向量数据库 ChromaDB 技术**
  - **2.5.1 向量表示与语义检索**  
    语义检索通常将文本通过嵌入模型编码为高维向量。文本之间的语义相似度可通过余弦相似度等指标衡量，从而实现“用语义找内容”，解决关键词检索对同义表达不敏感的问题。  
  - **2.5.2 ChromaDB 概述**  
    ChromaDB 是面向向量存储与检索的数据库系统，支持向量写入、相似度检索、元数据过滤等能力，适合构建领域知识库的语义索引。  
  - **2.5.3 在本系统中的作用**  
    本系统将合同条款与法律条文切分为片段后进行向量化，并写入 ChromaDB。用户提问时，系统在向量库中检索最相关的若干片段，并将其作为“可依据的上下文”提供给大语言模型生成回答，实现可解释的法律与合同条款引用。
- **2.6 LangChain 与检索增强生成（RAG）技术**
  - **2.6.1 LangChain 概述**  
    LangChain 是面向大语言模型应用开发的框架，提供提示词模板、链式编排、向量检索集成、对话记忆等组件，能够降低构建 LLM 应用的工程复杂度。  
  - **2.6.2 RAG 技术原理**  
    RAG（Retrieval-Augmented Generation）通过“先检索、后生成”的方式，在生成阶段为大语言模型提供高相关度的外部知识，从而提升回答的事实性与可追溯性。其典型流程包括：  
    - 文档采集与清洗（合同、法律条文）；  
    - 文本切分（按条款/章节/段落切片）；  
    - 嵌入向量化与索引构建；  
    - 用户提问→向量检索召回候选片段；  
    - 构造提示词并将“问题 + 召回片段”输入模型生成回答；  
    - 输出答案与引用来源（合同条款/法律条文）。  
  - **2.6.3 RAG 在本系统中的价值**  
    合同与法律属于高风险场景，回答需要“有依据”。RAG 可显著降低模型幻觉概率，并使回答能够附带引用条款与条文，提高可解释性与可审计性。
- **2.7 大语言模型与 API 调用技术**
  - **2.7.1 大语言模型概述**  
    大语言模型（LLM）基于海量语料训练，具备自然语言理解、信息抽取、总结归纳与文本生成能力。在垂直领域应用中，通常通过提示词工程、工具调用、RAG 等方式提升回答准确性与可控性。  
  - **2.7.2 API 调用与服务化接入**  
    工程实践中通常通过模型服务平台提供的 HTTP API/SDK 完成调用，后端负责：构造提示词、控制输出格式、处理错误与超时、记录调用日志等。  
  - **2.7.3 在本系统中的应用场景**  
    - **问题分类**：判断用户问题属于合同条款解释、风险识别，还是需要结合外部法律条文；  
    - **信息抽取**：从问题中识别可能的法律名称、条款范围、关键实体与约束条件；  
    - **答案生成**：综合合同片段与法律条文片段生成可解释回答，并输出引用依据；  
    - **结构化输出**：生成便于前端展示的结构（如“结论/依据/风险点/建议修改”）。  
  - **2.7.4 风险与控制措施（论文可写亮点）**  
    - 通过“检索增强 + 引用依据”降低幻觉；  
    - 通过提示词约束输出结构与措辞；  
    - 对关键回答增加免责声明与人工复核提示，降低误用风险。

## 第3章 系统分析
3.1  可行性分析  
3.1.1  技术可行性  
本系统采用Python作为主要开发语言，Python在数据处理、人工智能领域拥有丰富的生态库支持，能够高效实现系统核心功能。前端采用React+Ant Design构建，可提供直观流畅的用户交互界面；后端使用FastAPI框架，基于Python异步特性支持高并发接口调用，保证系统响应性能。数据库选用MySQL存储用户信息、合同信息等业务数据，ChromaDB存储合同与法律文本向量索引。通义千问大语言模型通过API接入，结合LangChain框架实现RAG问答流程，有效降低开发复杂度；句法分析借助LTP工具完成，文本对比使用Python内置的difflib库。上述技术栈成熟稳定，社区活跃，开发文档丰富，技术实现路径清晰可行。

3.1.2  经济可行性  
本系统开发阶段全部采用开源技术栈（React、FastAPI、MySQL、ChromaDB、LTP等），无需商业软件授权费用；大语言模型选择了阿里云的通义千问大模型，提供了免费额度的API服务，也可以选择在本地部署开源模型，调用成本可控。整体开发与运行成本在可接受范围内，具备经济可行性。

3.1.3  市场可行性  
本作品的目标用户主要针对非法律专业的普通公众，此类用户的法律素养水平相对较低，对合同条款和法律术语理解程度一般。用户一般具有中等到高等的教育水平，但是由于缺乏深入的法律学术背景且平时接触合同的机会较少，对法律法规的理解相对基础。当面对合同签署等问题时（如入职签约、租房合同、服务协议等），用户往往有意愿提升自身法律意识，希望完整地了解合同内容，以此提高在合同签署中的主动性和决策水平。在应用场景方面，本系统可覆盖个人签约前的合同解读、合同版本对比审核、法律条文检索学习等多种场景，应用范围广泛。随着社会法治意识的增强和企业合规需求的提升，市场对智能化合同辅助工具的需求持续增长，本系统具有明确的应用价值和市场前景。

- **3.2 功能需求分析**
  - **3.2.1 功能性需求（按核心模块组织）**
      （1）用户管理功能
系统首先需要提供完善的用户管理功能，包括用户注册、登录与身份认证等基本操作。新用户通过注册接口提交用户名、邮箱、密码等信息，经后端校验后写入用户表；已注册用户通过登录功能获取访问令牌（Token），后续所有涉及合同内容和问答历史的接口调用均需在认证通过的前提下进行。系统还应支持用户基本信息的查询与修改，为后续扩展角色权限、操作审计等功能预留空间。通过用户管理模块，可以确保不同用户的数据彼此隔离，避免合同内容和咨询记录在用户之间泄露。
（2）合同管理功能
合同管理模块是系统的基础功能之一，主要负责合同文件的全生命周期管理。系统应支持用户上传多种格式的合同文件，对上传的文件进行合法性校验（如文件大小、类型等），并在服务端进行持久化存储。合同上传完成后，后端需要对文件内容进行文本解析与预处理，抽取出纯文本并进行格式规范化，为后续的分段切分、语义向量化和长难句识别提供原始数据支撑。同时，系统应提供合同列表浏览、按名称或上传时间检索、查看合同详情以及删除无效合同等功能，使用户能够方便地管理自己上传的合同文档。
（3）知识图谱提取功能
为帮助用户从结构化视角理解合同内容，系统需要具备从合同文本中抽取知识图谱的能力。具体而言，系统应能够基于预先设计好的合同类型模板，从合同中识别出关键实体（如合同当事人、标的物、金额、期限等）及其之间的语义关系，并将这些信息构建为“主语−谓语−宾语”的三元组形式。抽取得到的三元组应与对应合同及条款位置进行关联，并存入后台数据库中。前端基于这些三元组数据，以力导向图等方式进行可视化展示，使用户能够从全局视角快速把握合同的核心结构和权利义务关系。
（4）长难句解析功能
合同文本中普遍存在句式复杂、嵌套较多的长句，普通用户难以直接理解其法律含义，因此系统需要提供长难句解析功能。该功能应能够对合同全文进行句子级扫描，结合句子长度、从句数量和句法复杂度等指标识别出潜在的长难句，并将这些句子作为候选进行进一步处理。对于识别出的长难句，系统应调用大语言模型生成通俗易懂的“大白话”解释，必要时还可以进行分句重写和关键术语说明。最终，长难句与其解释结果应一同存储，并在合同详情页面中以高亮和弹出说明的方式呈现，帮助非专业用户准确把握条款含义。
（5）智能问答功能
智能问答模块是本系统的核心功能之一，需支持基于具体合同的问答以及不绑定合同的通用法律问答两种模式。在合同问答模式下，用户先选择一份已上传的合同，再以自然语言提出问题，系统需要自动在相应合同片段及法律条文中检索相关内容，综合大语言模型生成结构化的解答；在通用问答模式下，系统则主要基于法律知识库回答与合同无关的法律常识问题。无论哪种模式，问答结果中都应尽量给出明确结论、关联依据条款及法律条文，并提示潜在风险与建议操作。同时，系统应记录用户的提问内容、回答结果和引用依据，形成多轮对话历史，以便用户后续查阅和比对。
（6）合同对比功能
在实际业务场景中，同一份合同往往会存在多个版本，不同合作方之间的合同文本也可能略有差异，因此系统需要提供合同对比功能。该功能要求用户能够任选两份合同（或同一合同的两个版本），系统对其条款进行匹配和差异分析。一方面，可以基于条款标题、编号进行直接匹配；另一方面，对于缺乏明显编号对应关系的条款，可通过文本相似度计算进行模糊匹配。完成条款级匹配后，系统应识别出新增条款、被删除条款以及内容发生修改的条款，对这些差异内容进行高亮展示，并在必要时调用大语言模型给出语义层面的“变化说明”和可能影响，辅助用户快速理解不同版本之间的实质性差异。
  - **3.2.2 非功能性需求**
    - 性能：在典型长度的合同上，问答响应时间控制在秒级至十秒级范围内，并在第6章通过测试数据进行验证；
    - 可用性：前端界面简洁、交互流畅，例如在问答结果中可直接看到对应合同条款和法律条文的引用片段；
    - 安全性：基于 JWT 的访问控制，合同文件和问答记录只对所属用户可见，避免数据泄露。
- **3.3 业务流程分析**
  - **3.3.1 用户注册登录流程**
    - 用户通过前端界面提交注册信息，后端完成校验并存储用户数据；
    - 登录时通过用户名和密码获取 Token，之后的接口访问携带 Token 完成鉴权。  
  - **3.3.2 合同上传与解析流程**
    - 用户选择本地合同文件上传；
    - 后端将文件存储到服务器，并解析出合同文本；
    - 对文本进行清洗与切分，将片段向量化并写入向量库；
    - 合同元数据和索引信息写入数据库，后续可用于问答和对比。  
  - **3.3.3 合同知识图谱抽取流程**
    - 在合同解析完成后，调用知识图谱提取服务；
    - 利用大语言模型按照预定义模板从合同中抽取实体与关系三元组；
    - 将三元组存入知识图谱表中；
    - 前端通过接口获取三元组数据并以力导向图等方式展示。  
  - **3.3.4 长难句解析流程**
    - 对合同文本进行遍历，识别候选长难句（结合句法复杂度或长度阈值）；
    - 将候选句子发送给长难句解析服务，由大语言模型生成通俗化解释；
    - 将原句与解释结果一起存储，并与合同及位置索引关联；
    - 前端在合同详情中高亮长难句并展示对应解释。  
  - **3.3.5 合同问答流程**
    - 用户在前端选定某一份合同并提出问题；
    - 后端根据问题内容确定检索范围（仅合同或合同+法律）；
    - 在向量库中检索与问题最相关的合同片段和法律条文片段；
    - 将检索到的片段整理为上下文，构造 Prompt 调用大语言模型生成答案；
    - 答案与引用片段、合同ID、用户ID一起存入对话历史表；前端展示答案并支持查看引用依据。  
  - **3.3.6 合同对比流程**
    - 用户在前端选择两份合同或同一合同的两个版本；
    - 后端对两份合同的条款做匹配（可基于标题、编号或文本相似度）；
    - 对匹配结果计算差异，标注新增、删除、修改、相同等类型；
    - 对关键差异内容可进一步调用大语言模型进行语义解释与确认；
    - 前端以对比视图展示差异点，帮助用户快速理解不同版本之间的变化。

- **4.1 总体架构设计**

  - 画出系统总体架构图：前端页面、后端服务、向量库、LLM 服务之间的调用关系。
- **4.2 功能模块设计**
  - 用户与权限模块：
    - 认证路由：登录、注册、获取当前用户信息等接口设计；
    - JWT 生成与校验机制；
    - 权限校验：确保当前用户只能访问自己上传的合同和对应的问答记录。  
  - 合同管理模块：
    - 合同上传接口：文件接收方式、存储策略和大小限制；
    - 文本解析与分块策略：对不同格式的合同进行文本抽取，并按条款/段落切分以支持向量化和长难句识别；
    - 合同列表、详情与删除接口设计。  
  - 知识图谱模块：
    - `knowledge_graph_service` 核心流程：以合同文本为输入，调用 LLM 按模板输出实体-关系三元组，再持久化到数据库；
    - 与前端 `KnowledgeGraph` 组件的数据接口：提供适合力导向图展示的节点和边数据结构。  
  - 长难句解析模块：
    - `sentence_parser_service` 的识别规则/算法：基于句法复杂度、句长、从句数量等指标识别长难句；
    - 解析结果结构化设计：包含原句、通俗化解释、所属合同ID、位置索引等信息。  
  - 智能问答模块（合同问答 + 通用问答）：
    - 基于后端问答逻辑（如 `qa.py`）描述如何区分 `contract_only` 与 `contract_and_law`；
    - 合同向量库和法律条文向量库的检索策略：TopK 设置、过滤条件（合同ID、法律名称等）；
    - Prompt 构造策略：system / user prompt 的整体结构和约束，强调引用条款与条文、输出结构化答案。  
  - 合同对比模块：
    - 条款映射设计：按条款标题/编号或文本相似度进行条款级匹配；
    - 差异类型设计：新增、删除、修改、相同等；
    - 前端展示方式：左右对比视图、差异高亮与语义说明区域。
- **4.3 系统用例设计**
  - 列出主要用例：上传合同、解析合同、提出合同相关问题、发起通用法律问答、查看历史对话、发起合同对比等；
  - 绘制用例图和用例描述，说明用户与系统间的主要交互。
- **4.4 数据库与数据模型设计**
  - 概念结构设计（E-R 图）：包括用户、合同、知识图谱三元组、长难句、对话会话和消息等实体；
  - 逻辑结构设计（数据表）：
    - `users`：用户基本信息；
    - `contracts`：合同元数据（文件名、路径、上传时间、所属用户、解析与索引状态等）；
    - `knowledge_graph`：合同知识图谱三元组（contract_id、subject、predicate、object、confidence、position 等）；
    - `long_sentences`：长难句解析结果（contract_id、sentence_text、explanation、position 等）；
    - `conversations` / `messages`：对话历史（用户ID、合同ID、问题、答案、时间戳、会话ID等）。
- **4.5 接口设计**
  - 认证相关接口：`/api/auth/*`（登录、注册、获取当前用户信息），包括请求方法、URL、参数和返回字段；
  - 合同管理接口：`/api/documents/*`（上传、列表、删除、详情）；
  - 向量搜索与 RAG 接口：`/api/search/*`、`/api/documents/{contract_id}/qa`（合同问答）；
  - 知识图谱接口：`/api/knowledge-graph/{contract_id}`、`/api/knowledge-graph/{contract_id}/extract`；
  - 长难句接口：`/api/contracts/{contract_id}/long-sentences` 及触发解析的相关接口；
  - 合同对比接口：如 `/api/contracts/compare`，包含对比合同ID参数和差异结果结构说明。
- **4.6 前端界面与交互设计**
  - 页面划分与路由：
    - 合同管理页面（`Contracts`）：合同上传组件、合同列表展示与基本操作；
    - 合同详情页面（`ContractDetail`）：知识图谱视图、文档预览、长难句列表与解释区域、绑定当前合同的问答区域；

    - 历史对话页面（`Conversations`）：展示对话列表与具体对话详情。  
  - 主要交互流程与界面布局：
    - 从合同管理页面上传并选中合同，跳转至详情页进行查看与提问；
    - 在详情页中切换知识图谱、长难句、问答等子区域；
    - 在问答页面输入问题、查看回答与引用条款；
    - 在历史对话页面回看某次会话的完整问题与回答记录。

## 第5章 关键技术与系统实现
- **5.1 系统开发环境与部署方案**
  - **5.1.1 开发环境**
    - 后端：Python + FastAPI；
    - 前端：React + Ant Design；
    - 数据库：MySQL（业务数据）+ ChromaDB（向量索引）；
    - 模型服务：通义千问（DashScope API）或等价大模型服务。  
  - **5.1.2 部署方案（可按你实际部署改写）**
    - 前端构建后以静态资源方式部署；
    - 后端以 ASGI 服务方式部署（如 Uvicorn/Gunicorn）并提供 API；
    - MySQL 与 ChromaDB 作为基础服务部署在本机或服务器；
    - 模型通过 API Key 方式接入第三方模型服务。  
  - **5.1.3 配置与参数管理（论文可写工程点）**
    - 将数据库连接串、模型 Key、TopK、超时等参数写入配置文件/环境变量；
    - 对外提供统一的错误码与错误信息，便于前端提示与排查问题。
- **5.2 合同与法律文本预处理与切分实现**
  - **5.2.1 文本解析与清洗**
    - 统一编码与换行格式；
    - 清除多余空白、页眉页脚、无意义符号（视数据情况）；  
  - **5.2.2 法律条文切分实现**
    - 基于法律文本常见结构（如“第X条”）进行规则切分；
    - 保留条号、标题、正文，形成结构化片段；
    - 将“法律名称/条号/章节”等作为元数据，便于检索过滤与引用展示。  
  - **5.2.3 合同条款切分实现**
    - 采用段落切分结合规则（如编号、标题）形成片段；
    - 保留片段序号与位置，支持回答引用时定位到原文。  
  - **5.2.4 嵌入向量生成**
    - 选择合适的嵌入模型将片段编码为向量；
    - 控制片段长度与重叠策略（chunk size / overlap）以兼顾召回与可读性。
- **5.3 法律知识库与向量检索实现**
  - **5.3.1 向量库 Schema 与元数据设计**
    - 合同片段元数据：合同ID、片段编号、标题/章节、来源文件等；
    - 法律片段元数据：法律名称、条号、章节、发布日期（可选）。  
  - **5.3.2 索引构建与批量导入**
    - 对法律文本支持批量导入与增量更新；
    - 对合同文本支持“上传即入库”，并可在删除合同时同步删除向量索引（可选）。  
  - **5.3.3 检索与重排策略（可写“提升效果”）**
    - 向量召回 TopK；
    - 基于元数据过滤（如限定某一部法律）；
    - 对召回片段做去重、长度裁剪与排序，提升输入上下文质量。
- **5.4 智能问答与 RAG 流程实现**
  - **5.4.1 问题分类实现**
    - 通过大模型或规则对问题进行分类：合同相关/法律相关/综合；
    - 分类结果决定检索范围（仅合同库、仅法律库或两者同时）。  
  - **5.4.2 法律名称与条文抽取实现**
    - 在“需要法律依据”的场景下，从用户问题中抽取法律名称或关键词；
    - 将抽取结果用于向量库元数据过滤，提高召回精度。  
  - **5.4.3 提示词构建与上下文组织**
    - 将检索得到的合同片段与法律条文片段分区组织（避免混杂）；
    - 约束模型输出结构：结论、依据、风险点、修改建议、引用列表；
    - 设置“若证据不足则说明无法判断”的安全策略。  
  - **5.4.4 结果后处理与引用对齐**
    - 对模型输出进行结构化解析（JSON 或固定段落格式）；
    - 对引用条款/条文做规范化展示（如“某合同第X条”“某法第Y条”）；
    - 将引用映射回片段编号，支持前端定位与高亮。  
  - **5.4.5 多轮对话与历史记录（可选）**
    - 保存会话上下文与关键引用；
    - 控制历史长度，避免上下文膨胀与成本上升。
- **5.5 前端界面与交互实现**
  - 合同上传与管理界面实现（React + Ant Design）。
  - 合同预览与条款展示界面设计。
  - 智能问答对话界面实现（问题输入、回答展示、引用高亮等）。
  - 用户交互体验优化措施（加载状态、错误提示、历史记录等）。
